# virtual-screen-reader Gap Analysis & Adoption Assessment

**Decision:** adopt [guidepup/virtual-screen-reader](https://github.com/guidepup/virtual-screen-reader) (VSR) directly as the **fifth execution mode of `a11y-test` — component/unit-level screen-reader assertions** in the consuming project's own test suite (Vitest/Jest with jsdom, a real browser via its ESM build, or Storybook play functions — all exercised below; Storybook lane added 2026-07-13). Adoption shape is the `eslint-plugin-jsx-a11y` pattern, not the keyboard-a11y-tester pattern: an npm devDependency installed in the project under test, exact-pinned at `0.32.1`, with snippet templates and calibration rules living in the skill. Do not vendor. **Done (2026-07-11): all four phases executed** — Phase 1 routing, Phase 2 fixture cross-validation (gate passed; record at `evals/results/virtual-screen-reader/`), Phase 3 critic/contract evidence wiring, Phase 4 lifecycle integration.

Assessed 2026-07-11 at v0.32.1 (npm). MIT license, author Craig Morten (guidepup org). Hands-on validation performed in plain Node 24.13 + jsdom 29.1.1, in a real Vitest 4.x jsdom harness, and in real Chromium via the ESM browser build (probes and results below), plus a spot-validation against one critic fixture's planted defects. Reviewed by proposal-critic 2026-07-11 (verdict REVISE); this revision executes its two run-before-you-write preconditions (real-harness and browser-mode runs) and incorporates all findings.

**Already in our stack, transitively:** VSR is the announcement-emulation, live-region-monitoring, and reading-order-census engine inside keyboard-a11y-tester — a *runtime* dependency of KAT (`^0.32.1`, frozen by KAT's committed lockfile at our `0.5.0` pin; resolves to the same 0.32.1 assessed here). This assessment does not change that route; it adds a *direct* route for the layer KAT cannot serve: components under development, before any URL exists. The flip side of the shared engine is a shared point of failure — see Risks.

## What It Is

A screen-reader **simulator as a library**: it walks the accessibility tree of any DOM (jsdom in unit tests, or a real browser via its ESM build — the path KAT uses), emits spoken phrases (`"button, Save document"`, `"heading, Settings, level 2"`), captures live-region announcements with politeness prefixes (`"polite: Draft saved"`), and exposes screen-reader quick-key emulation as an API: `moveToNextHeading`, `moveToNextLandmark`, `moveToNextForm`, per-level heading jumps, `jumpToErrorMessageElement` (aria-errormessage), aria-flowto reading-order commands. Tests assert on `spokenPhraseLog()` like any other unit assertion.

Spec-anchored, not vibes-anchored: implements ACCNAME 1.2, CORE-AAM 1.2, HTML-AAM, ARIA in HTML, WAI-ARIA 1.2 (plus SVG/graphics/dpub AAMs), and runs against the W3C Web Platform Tests with published pass/fail counts (396 suites passing, 81 failing tests, 338 skipped, with per-skip reasons in-repo). Dependencies are the respected accname stack: `dom-accessibility-api`, `@testing-library/dom`, `@testing-library/user-event`, `html-aria`. Its own docs state the same rule our KAT calibration rule 5 encodes: it augments real screen-reader testing, never replaces it.

**Upstream maturity, measured (2026-07-11):** created 2023-05; 141 stars; ~76k npm downloads/month; MIT. Effectively single-maintainer (Craig Morten under two accounts; dependabot is the second-largest committer). Last feature release 0.32.1 on **2025-05-19 — ~14 months ago**; since then activity is dependabot servicing and a cosmetic README edit (2026-06-22). The guidepup org as a whole is alive (pushes across `guidepup`, `jest`, `voiceover`, `setup`, `guidepup-playwright` in late June/July 2026). 7 open issues/PRs, all long-lived feature asks. Read: stable and serviced, not fast-moving — the opposite risk profile from KAT at its adoption (2-day-old repo, daily motion).

## Gap Analysis — What It Adds

1. **Announcement evidence at the component/unit layer.** Today the only machine evidence for "did this live region actually announce?" is KAT — which needs a deployed URL, a cloned runner, Chromium, and a journey. During `implement → test` on a component there is nothing: `.spec.js` asserts ARIA attributes, and §1's own mandate says attributes alone prove nothing. VSR is the aria-live analogue of that mandate, runnable per-PR in CI in milliseconds. This is the cheapest point in the lifecycle to catch the defect class our toast/async-form/infinite-scroll fixtures plant.
2. **Reading-order and SR-reachability assertions as unit tests.** "An SR user can reach the error summary by heading navigation" / "the census reads price before the buy button" — the quick-key command API makes these assertable per component. Currently only KAT's page-level census covers reading order, post-deploy.
3. **Accessible-name ground truth in CI.** Accname computation per ACCNAME 1.2 (WPT-tested upstream) catches missing names (probe A) and wrong names (probe A2 — `aria-label` contradicting visible text, a class axe's label rules under-specify), in the same test file as the component's other unit tests.
4. **A stack we already run daily via KAT.** Same package, same resolved version line today. Note this is *familiarity*, not *added safety*: as an npm devDependency, VSR and its four `^`-ranged transitive deps (`dom-accessibility-api`, `@testing-library/dom`, `@testing-library/user-event`, `html-aria`) enter the consuming project's lockfile — a slightly larger supply-chain touchpoint than KAT-as-clone, whose tree stays in its own checkout. Dev-only in both cases.

## Gap Analysis — What It Does NOT Cover (and what keeps covering it)

- **Keyboard operability — hard boundary.** VSR interactions run through bundled `@testing-library/user-event`: **synthetic events**. Under our mandatory real-keyboard-events rule, VSR output is *never* keyboard-operability evidence. A VSR `press()` that "works" proves nothing about a real browser. Keyboard evidence stays with `.spec.js`, agent-browser, KAT.
- **Anything visual.** jsdom has no layout: no focus-indicator measurement, no reflow, no contrast, no target size. Visual lanes unchanged (Playwright, KAT's pixel measurement).
- **Journeys and pages.** URL + goal → findings remains KAT's lane. VSR emits no findings at all — it is an assertion primitive; the test author supplies the judgment.
- **Shadow DOM components** (probe E) — for web-component UIs (Lit, Shoelace, Drupal SDC compiled to custom elements with shadow roots), VSR walks are silently partial. See calibration rule 2.
- **Known unsupported ARIA**: `aria-busy` (#36 — note: our `loading-state-missing-aria-busy` fixture is therefore out of VSR's scope), feed role (#13), Shift+`press()` capitalisation quirk (#8).
- **Real AT behavior.** Emulated ≠ NVDA/JAWS/VoiceOver. §6 manual protocol survives untouched — same as KAT calibration rule 5.

## Calibration Rules

Rules 1–4 are measured behavior from the probes above (environments noted); rule 3 contains one explicitly-labeled non-measured limb; rule 5 is consumption discipline.

1. **Never walk-to-end-of-document on pages containing `aria-modal="true"`** *(measured, jsdom)*. The cursor traps in the modal *by design* — the loop is intended containment, not a bug. Scope `container` to the component under test, and bound every walk loop with a max-step guard.
2. **A silent or short walk is not a clean pass — check for shadow roots first** *(measured, jsdom)*. Open shadow DOM content is invisible to VSR (#182). A web component that announces nothing may be perfectly accessible. If `element.shadowRoot` exists anywhere under the container, VSR evidence is partial: route to KAT/browser testing instead.
3. **Mount-with-content live regions read as silent — including correctly-fixed ones** *(measured in jsdom AND Chromium)*. VSR announces mutations *inside* existing live regions (text changes, child insertions, mount-empty-then-fill) but not insertion of a pre-populated `role="alert"` element. Removal behavior splits on `aria-atomic` *(measured, Phase 2 sweep)*: clearing a non-atomic region is silent; clearing an `aria-atomic="true"` region announces an empty `"polite: "` phrase — treat an empty polite entry as a region-clear marker, not noise. The measured limb ends there. The interpretive limb — that real screen readers are also inconsistent on pre-populated alert insertion, which is why robust toast guidance uses a persistent live container — is domain knowledge (a11ysupport.io-class sources), **not probed here**; do not cite it as measured. Consequences: assert announcements via the persistent-container pattern; a silent log after mount-with-content of an alert is **inconclusive**, not a failure of the fix; buggy-toast silence is valid defect evidence only alongside the structural fact (no role/aria-live present).
4. **Fake timers wedge the singleton — hard incompatibility** *(measured, Vitest)*. Under `vi.useFakeTimers()`/`jest.useFakeTimers()`, `start()` resolves but log reads hang forever, and the wedged state cascades hangs into every later test in the file, teardown included. Never enable fake timers in a file that runs VSR. Components whose behavior needs fake timers (auto-dismiss toasts) get their announcement assertions in a separate real-timer file — which the persistent-container pattern makes natural (assert the announcement on show; the 3-second dismiss is a separate concern). With real timers, no arbitrary sleep is needed: announcements are readable after a microtask yield (`await Promise.resolve()`; use your runner's `waitFor` for belt-and-suspenders).
5. **Cite the VSR version in every piece of evidence** *(discipline)*. Both consumption routes are frozen (direct exact-pin; KAT's committed lockfile at our pinned release) and live in separate dependency trees, so skew arises only when either side deliberately upgrades — this rule makes it *detectable*, not prevented. Residual risk is low because the calibrated behaviors above are structural, unlikely to shift within a patch line — but re-verify rules 1–4 on any version bump, either side.

## Risks & Uncertainty

- **Concentration: one engine now underpins two of five execution modes.** VSR is a runtime dependency of KAT *and* the proposed direct lane. A single upstream failure — abandonment, a breaking transitive-dep change, a Node incompatibility — degrades automated screen-reader/announcement/census evidence in both lanes at once, and the org-level alternatives (`@guidepup/guidepup`, `@guidepup/jest`) are the same maintainer, so they are correlated fallbacks, not diversification. **The capability floor if the engine dies is the §6 manual AT protocol** — that is the honest fallback, and it is why §6 remains a shipping gate rather than a legacy section. What bounds the blast radius: MIT + exact-pin means the assessed 0.32.1 keeps working as long as it installs (failure mode is erosion-on-drift as Node/deps move, not sudden disappearance), the surface is small enough to fork, and committed spoken-phrase logs remain readable evidence with no tool at all.
- **Single maintainer, slow release cadence.** 14 months since a feature release; the shadow DOM fix has sat unmerged since April 2026. Counterweights: 76k downloads/month, an active org, spec-anchored behavior that ages slowly, dependabot servicing. Escape hatch for the *lane* is genuinely cheap (plain test code, deletes cleanly); escape hatch for the *capability* is the manual protocol above — cheap-lane-removal must not be mistaken for cheap-capability-replacement.
- **False confidence / manual-AT atrophy.** The failure mode where teams treat green VSR assertions as "screen reader users are fine." Mitigation is the same framing that already works for KAT: emulation evidence is labeled as such; §6 remains a shipping gate.
- **Rule-3 subtlety will bite someone.** A developer who fixes the toast and sees a still-silent log will conclude the fix failed or the tool is broken. Now measured in both environments and shipped as a Phase 1 template (persistent container) rather than a caution; the residual risk is developers who skip the template. Phase 2 item 6 keeps our own fixtures from teaching the conflicting lesson.
- **Where confidence drops:** Jest is unexercised — both jsdom mode (expected to match Vitest) and its fake-timer implementation (the rule is written harness-generic because the mechanism — faked ticks starving VSR's internal waits — is harness-independent, but Jest-specific behavior is unmeasured). VSR's AOM opt-in mode is unexercised. The Storybook validation is one framework (React 19 via react-vite) and one runner generation (10.4.6 + addon-vitest); older Storybook 8 setups using `@storybook/test-runner` are expected to work (upstream's example is SB8) but were not run here.

## What This Does Not Claim

- Not keyboard-operability evidence, ever — synthetic events (user-event) are on the wrong side of our mandatory real-events rule.
- Not a replacement for KAT (journeys/URLs), §6 manual AT, axe, or the visual lanes; not a findings emitter — the test author's judgment supplies severity, the contract mapping supplies structure.
- Not usable on open-shadow-DOM components until upstream #182 lands.
- Not verified in Jest (jsdom or fake timers — Vitest only) or in Storybook 8's `@storybook/test-runner` (the 10.x addon-vitest lane is what ran); AOM opt-in unexercised; not a claim that emulated announcements match any specific real screen reader.
- Not a new cross-model benchmark lane, and not part of the Ollama portability surface (a11y-test is already flagged non-portable).
- No independent security/supply-chain audit performed — and note the surface differs from KAT: VSR's transitive deps enter consuming projects' lockfiles (dev-only), where KAT's stay in its own clone. That audit becomes mandatory if vendoring is ever revisited.
